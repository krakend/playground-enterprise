02 Avoid duplication

Multiple teams call the same AI services independently and pay for the same tokens more than once.

"Central IT sets the guardrails; each team gets its own token budget. Finance sees the spend per team — in one place."

Token quota/budget is enforced per role in Redis (moderator 10k/day, reader 1k/day). Run this to spend against the moderator budget:

TOKEN=$(curl -s -X POST http://localhost:8085/realms/krakend/protocol/openid-connect/token \
  -d grant_type=password -d client_id=playground \
  -d client_secret=HA7yGHBNm5IBBSJYsSgtmJgnqdeeItrw \
  -d username=moderator -d password=moderator \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['access_token'])")

curl -s -X POST http://localhost:8080/llm-quota \
  -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' \
  -d '{"contents":"List 3 differences between REST and GraphQL"}'
Semantic cache — roadmap (KrakenD 3.0): the same or a semantically-equivalent prompt returns a cached answer with zero tokens. First call hits the LLM and is stored; the second matches on meaning, not exact text, and never reaches the provider. Shipping in KrakenD 3.0.

03 Scalability

Spiky, unpredictable AI traffic across heterogeneous platforms, with unknown future load.

"Scalability is a Kubernetes concern, not an AI Gateway concern. You add replicas; there is no shared state to synchronize, no master node."

Stateless binary + config. Redis is only for quota persistence, never in the routing path. Adding an instance is one line — same config, no sync:

# Illustration: each replica runs the identical config, no coordination.
# In Kubernetes this is a replica count; in Compose, another service:
#   krakend_ee_2:
#     image: krakend/krakend-ee:2.13-watch
#     volumes: [ "./config/krakend:/etc/krakend" ]
# Same config in, gateway scales out.

04 Reusability

AI endpoints built in one platform can't be consumed by another without duplication. Define once, consume anywhere through a standard interface.

"You define the endpoint once. Any team, any platform calls the same URL. Switch providers next year and you change one config file, not every app."

Claim-based routing: the JWT role picks the model — the consumer never knows. Moderator → OpenAI, reader → Gemini:

TOKEN=$(curl -s -X POST http://localhost:8085/realms/krakend/protocol/openid-connect/token \
  -d grant_type=password -d client_id=playground \
  -d client_secret=HA7yGHBNm5IBBSJYsSgtmJgnqdeeItrw \
  -d username=reader -d password=reader \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['access_token'])")

curl -s -X POST http://localhost:8080/llm-conditional \
  -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' \
  -d '{"contents":"In one sentence, what is an API gateway?"}'

06 Security — AI defense

Prompts can leak secrets or carry injection/jailbreak attempts. Guardrails run at the gateway, before anything reaches the model.

"PII, secrets and prompt-injection are stopped at the edge — in line, with no change to the apps behind it."

Governance, woven throughout

  • Credential isolation — provider API keys are injected at the gateway; no app team ever sees them. Rotate in one place.
  • Decentralized policy — each team's rate limits, quotas and routing live in config. Central IT sets guardrails; solution owners operate within them.
  • Audit trail — every call logged, every token counted, per team. "How much are we spending on AI and who owns it" is already on the dashboard.
